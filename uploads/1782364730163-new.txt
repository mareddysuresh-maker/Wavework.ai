## 1. PRODUCT VISION

WaveWork is a workspace-centric project management and team
collaboration platform, modeled closely on **ClickUp's** core
information architecture:

```
Workspace
  → Spaces
      → Lists
          → Tasks
              → Subtasks, Comments, Attachments
  → Channels (group chat)
  → Direct Messages (1:1 chat)
  → Member Directory
  → Activity Logs
  → Workspace Settings
```

Like ClickUp, **Spaces are the primary organizing unit** beneath the
Workspace — they represent departments, teams, or initiatives
(e.g. "Development", "Design", "Marketing"), each containing Lists,
which contain Tasks. Every workspace member can see every Space (shared
visibility), while the ability to *manage* structure (create/edit/
delete Spaces and Lists) is restricted by role.

WaveWork is **not multi-tenant in this version** — each deployment
instance operates as a single isolated Workspace per the architecture
spec below — but the data model must be built so that multi-workspace
support can be layered on later without restructuring (every entity is
already workspace-scoped).

On top of the ClickUp-style project hierarchy, WaveWork adds:

- A unified, role-based **sidebar** that adapts per user role.
- Real-time **channels and direct messages**, similar to Slack, fully
  integrated with task mentions and notifications.
- A private **Personal Space** per user (personal tasks, scratchpad,
  whiteboard) that no one else — including admins — can see.
- A personal **File Hub** that aggregates files received via task
  assignments without duplicating storage.
- A centralized **Activity Log** for full auditability.
- A **3-role permission system** (Super Admin, Admin, Employee) with
  strict backend enforcement, detailed fully in Section 5.

## 2. MANDATORY TECHNOLOGY STACK

Do not substitute any of the following. Use the **MERN stack
philosophy** with TypeScript on the frontend only:

| Layer | Technology | Notes |
|---|---|---|
| Frontend | **React** (with **TypeScript**) | Functional components, hooks only. No class components. |
| Frontend build tool | **Vite** | Fast dev server, HMR, optimized production builds. |
| Frontend styling | **Tailwind CSS** | Utility-first; no inline styles, no CSS-in-JS libraries. |
| Frontend animation | **Framer Motion** | For transitions, drag-and-drop visual feedback, modals. |
| Frontend state | **React Context + hooks**, or **Zustand** for global state | Avoid Redux unless complexity demands it. |
| Frontend data fetching | **TanStack Query (React Query)** | Caching, retries, optimistic updates. |
| Backend runtime | **Node.js** | LTS version. |
| Backend language | **JavaScript** (plain Node.js, **not** TypeScript) | Per requirement: backend stays JavaScript. |
| Backend framework | **Express.js** | REST API layer. |
| Database | **PostgreSQL** | Relational integrity for workspace/role/task relationships. |
| ORM | **Prisma** | Schema-first, migrations, type-safe queries. |
| Real-time | **Socket.io** | Chat, notifications, live task updates. |
| Real-time scaling | **Redis** (via `socket.io-redis-adapter` / pub-sub) | Used specifically for the **chat system** — horizontal Socket.io scaling and presence/online-status tracking across multiple server instances. |
| File storage | **MinIO** (S3-compatible object storage) | Self-hosted, free, S3 API-compatible — used for all file/attachment/PDF/document uploads. |
| Authentication | **JWT** (access + refresh token pattern) | Stored in httpOnly cookies. |
| Password hashing | **bcrypt** | Never store plaintext passwords. |
| Email | **Nodemailer** with a free SMTP provider (e.g. Gmail SMTP or a free-tier transactional email API) | Used for invitations and notifications. |
| Validation | **Zod** (shared-shape validation; duplicated in JS on backend, TS on frontend) | Validate every request body server-side regardless of frontend validation. |
| API documentation | **OpenAPI/Swagger** (`swagger-jsdoc` + `swagger-ui-express`) | Auto-generated from route annotations. |
| Testing | **Vitest** (frontend) + **Jest + Supertest** (backend) | Unit + integration coverage on permission logic above all else. |
| Containerization | **Docker + Docker Compose** | One container per service: frontend, backend, postgres, redis, minio. |
| Process management (prod) | **PM2** (or Node cluster module) | Multi-core utilization on a single host. |

**"High-end tech, free tools only" constraint:** every piece of
infrastructure above (PostgreSQL, Redis, MinIO, Socket.io, Prisma,
Nodemailer with free SMTP) has a genuinely free, self-hostable option
with no proprietary vendor lock-in. Do not introduce paid SaaS
dependencies (no Auth0, no paid Pusher/Ably channels, no paid S3 —
MinIO replaces S3 entirely for free).

## 2A. UI DESIGN SYSTEM (LOCKED — APPLY EVERYWHERE, NO EXCEPTIONS)

The product must look **sleek, clean, and calm** — closer to Linear or
ClickUp's own modern UI than to a generic admin-template dashboard. No
gradients-for-the-sake-of-it, no decorative illustrations, no
over-animated transitions, no mismatched icon sets. Every screen in
5A.5's sidebar-to-screen mapping must follow this system exactly so the
whole app feels like one product, not a patchwork of AI-generated
screens.

### 2A.1 Visual tokens

- **Typography:** one font family for the entire app — Inter (or the
  system font stack as a fallback). Two weights only: 400 (body) and
  600 (headings/emphasis). No more than 4 font sizes on any single
  screen (e.g. 12px meta text, 14px body, 16px section titles, 20px
  page titles).
- **Color:** one neutral gray scale (backgrounds, borders, secondary
  text) + exactly one accent color (buttons, active states, links,
  the unread-badge dot) + 4 semantic colors (success/green,
  warning/amber, danger/red, info/blue — used only for status pills
  like task priority or activity-log event type, never decoratively).
  Dark mode is not required for v1; build light mode only, correctly,
  rather than both poorly.
- **Spacing:** a single 4px-based spacing scale (4, 8, 12, 16, 24, 32,
  48). No arbitrary one-off padding values anywhere in the codebase.
- **Corners & elevation:** one border-radius value for cards/inputs/
  buttons (e.g. 8px), one for pills/avatars (full round). One shadow
  style for floating elements (modals, dropdowns) — flat, low-opacity,
  no heavy drop shadows.
- **Icons:** **one icon set only** — `lucide-react` — for every icon
  in the app (sidebar items, buttons, status indicators, empty states).
  Never mix icon libraries, never use emoji as functional icons. Each
  sidebar item in 5A.5 gets exactly one consistent icon used nowhere
  else for a different meaning (e.g. a checklist icon for "My Tasks"
  everywhere it appears, not a different icon in the sidebar vs. the
  page header).

### 2A.2 Motion — restrained, not decorative

Framer Motion is in the stack for **functional** motion only:

- Page/panel transitions: simple fade + 4px slide, under 200ms.
- Modal open/close: scale 0.98→1 + fade, under 150ms.
- Drag-and-drop (kanban board, list reordering): smooth follow with no
  bounce/spring overshoot.
- Toasts/notifications: slide in from the same corner every time.

No parallax, no scroll-jacking, no looping background animation, no
animated gradients. If a transition doesn't help the user understand
*what just changed*, it doesn't belong.

### 2A.3 Component consistency rules

- **One button component**, with exactly 3 visual variants (primary,
  secondary, destructive) and 2 sizes — used for every button in the
  app, including inside modals and forms. No screen invents its own
  button style.
- **One input/select/textarea style**, reused everywhere a form
  appears (task creation, workspace settings, invitations, comments).
- **One empty-state pattern**: icon + one line of text + one primary
  action, reused for "no tasks yet," "no channels yet," "no
  notifications," etc. — not a custom illustration per screen.
- **One avatar component** (initials-fallback circle), used in the
  sidebar, member directory, task assignee field, and chat — same
  size scale everywhere a given context calls for it.
- **One status-pill component** for task status, task priority, and
  invitation status — same shape, same semantic colors from 2A.1,
  different label text only.

### 2A.4 Layout discipline

- Sidebar width is fixed and identical across all three roles — only
  the *items inside it* change per 5A.5, never its width, font size,
  or icon style.
- Every list/table screen (tasks, members, activity logs, files) uses
  the same table component: same row height, same header style, same
  pagination control at the bottom — never a bespoke table per module.
- Every modal (Create Task, Create Channel, Invite Member, Create
  Space) uses the same modal shell (header + body + footer with
  Cancel/Confirm buttons in the same position) — only the body content
  differs.

### 2A.5 "No fancy things" — explicit exclusions

Do not build, and remove if AI-generated by default: animated
particle/gradient backgrounds, 3D tilt-on-hover cards, skeleton
loaders with shimmer effects beyond a simple pulse, custom cursors,
confetti/celebration animations, glassmorphism/frosted-blur panels,
or any illustration library (unDraw, etc.) for empty states. The bar
for every visual decision is: *does ClickUp or Linear actually do
this in their real product UI?* If not, don't add it here either.

## 4. SCALABILITY, PERFORMANCE & INFRASTRUCTURE REQUIREMENTS

WaveWork must be architected to handle **high concurrent traffic and a
large number of simultaneous real-time connections**, using only free,
self-hostable infrastructure. Apply the following:

### 4.1 Database performance
- Add indexes on every foreign key column and every column used in a
  `WHERE` filter (especially `workspaceId`, `assignedTo`, `status`,
  `createdBy`).
- Use Prisma's `select` to fetch only needed fields on list endpoints —
  never `findMany()` with implicit full-row selects on large tables.
- Use cursor-based pagination (not offset-based) for Tasks, Messages,
  Activity Logs, and Notifications, since offset pagination degrades
  at scale.
- Use PostgreSQL connection pooling (Prisma's built-in pool, tuned via
  `connection_limit` in the database URL) sized to the deployment's
  CPU count.

### 4.2 Caching
- Use Redis as a read-through cache for hot, infrequently-changing
  data: workspace settings, member directory listing, and channel
  metadata.
- Invalidate cache keys explicitly on writes (no time-based-only
  expiry for data that must stay consistent, e.g. role changes).

### 4.3 Real-time chat scaling (Redis)
- Socket.io must run behind the **Redis adapter**
  (`@socket.io/redis-adapter`) so multiple backend instances can share
  socket state and broadcast across instances — this is the specific,
  required use of Redis in this build.
- Use Redis to track **online/presence status** per user
  (`SET user:{id}:online` with TTL refreshed on heartbeat) rather than
  holding presence purely in-memory on one Node process.
- All chat and notification broadcasts must emit to a
  workspace-scoped Socket.io room (e.g. `workspace:{workspaceId}`),
  never a global broadcast, both for correctness (workspace isolation)
  and for performance (smaller fan-out per event).

### 4.4 File storage scaling (MinIO)
- All file uploads (task attachments, chat file-shares, profile
  pictures, workspace logos) go through MinIO, never local disk
  storage, so the backend remains stateless and horizontally
  scalable.
- Use pre-signed URLs for both upload and download where practical, so
  large file transfers don't proxy through the Express server's event
  loop.
- Store only the MinIO object key + metadata (size, mime type,
  uploader, workspace ID) in PostgreSQL — never the file binary.

### 4.5 Horizontal scaling readiness
- The backend must be stateless: no in-memory session storage, no
  in-memory socket-only state that isn't also mirrored in Redis. This
  allows running multiple backend container replicas behind a load
  balancer (see `infra/nginx/`).
- Use environment-variable-driven configuration throughout (no
  hardcoded ports, hosts, or secrets) so the same image can be deployed
  to any number of replicas.

### 4.6 Rate limiting & abuse protection
- Apply a Redis-backed rate limiter (e.g. `rate-limit-redis` with
  `express-rate-limit`) on authentication endpoints and on
  write-heavy endpoints (task creation, message sending) to protect
  against traffic spikes and abuse, since in-memory rate limiting
  doesn't work correctly across multiple backend replicas.

### 4.7 Background jobs
- Long-running or deferred work (invitation-expiry sweeps, activity
  log archival housekeeping) should run as scheduled jobs (`node-cron`
  is acceptable as a free, dependency-light scheduler) rather than
  blocking request/response cycles.

================================================================
CRITICAL IMPLEMENTATION RULES
================================================================

This is a **greenfield build** for this engagement — there is no prior
WaveWork codebase to preserve in this context. However, the rules below
were originally written for a refactor of an existing system, and their
underlying intent — minimal surface area, clean separation of concerns,
and strict backend enforcement — still governs how this fresh build
must be structured:

Only build files, routes, models, permissions, APIs, sidebar
configuration, task workflows, notifications, and UI components that
are directly required for the hierarchy described in this document.

Do not introduce speculative modules beyond what's specified here.

Do not couple unrelated features together.

Preserve clean module boundaries as the codebase grows.

Before implementing any feature:

1. Design the data model first.
2. Reuse existing patterns within this codebase once established
   (e.g. one consistent permission-check helper, reused everywhere).
3. Extend existing models rather than creating overlapping ones.
4. Touch the minimum number of files required for each feature slice.
5. Preserve backward compatibility within the app's own evolving
   schema once the first version ships (additive migrations
   preferred over destructive ones).

IMPORTANT — PERMISSION ENFORCEMENT LAYER:

Permission checks must be enforced BOTH:

- Frontend UI Level
- Backend API Level

Hiding buttons is NOT enough.

Every restricted action must also be validated server-side.

Employees must never gain access through:

- Direct API calls
- Browser DevTools
- URL manipulation
- Client-side bypasses

Backend permission validation is mandatory.

================================================================
PERMISSION CONFLICT RESOLUTION RULES
================================================================

If two sections appear to conflict, resolve in this order of priority:

1. Backend API Permissions take priority.
2. Role Restrictions take priority over UI visibility.
3. Workspace Isolation Rules take priority over all other rules.
4. Activity Logging requirements apply to all actions.
5. Security rules override convenience features.

Example: If a button is visible due to a UI bug but backend permission
denies the action, the request must fail.

DATABASE TRANSACTION RULES
================================================================

Critical multi-step operations must be executed atomically (Prisma
`$transaction`).

Examples:

- Workspace Creation
- Invitation Acceptance
- Role Changes
- Task Assignment
- Workspace Ownership Transfer

If any step fails:

- Rollback entire operation
- Prevent partial data creation

Example:

Workspace Creation
Create Workspace
→ Create Super Admin Membership
→ Create Default Notifications

If Membership creation fails: Workspace should NOT be created.

Purpose: Prevent inconsistent database state.

TASK VISIBILITY RULES
================================================================

Task visibility and task modification permissions are separate
concepts.

All workspace members can view tasks inside visible spaces.

Since all members can see all spaces, all members can also see the tasks
inside those spaces.

Example:

Workspace: Apple
Members: Karthik (Super Admin), Ravi (Admin), Sai (Employee), Rahul
(Employee)
Space: Development → List: Backend
Tasks: "Build Login API" (Assigned To Rahul), "Build Register API"
(Assigned To Sai)

Result: Sai can see Rahul's task. Rahul can see Sai's task. Visibility
is shared across the workspace. Permissions remain restricted.

Employees may view:

- Task Title
- Task Description
- Assignee
- Status
- Priority
- Due Date
- Progress Percentage
- Attachments
- Comments

Employees may comment on and participate in discussion on any task
they can see, regardless of who it's assigned to.

Employees may NOT:

- Edit tasks assigned to other users
- Delete tasks assigned to other users
- Reassign tasks assigned to other users
- Modify management-controlled fields on tasks assigned to other users

Employees may only modify tasks assigned to themselves, subject to the
Self-Assignment rules below.

Example:

Development Space → Backend List
Task: "Build Login API" — Assigned To: Rahul

Sai can see the task. Sai cannot modify Rahul's task.
Rahul can update the task because it is assigned to him.

----------------------------------------------------------------
TASK VISIBILITY IMPLEMENTATION REQUIREMENT
----------------------------------------------------------------

Do NOT filter task visibility (read queries) by Assigned User ID.

Task read queries must be scoped by Workspace ID only — every
workspace member's task list/board/detail queries return every
non-deleted task in the workspace, not just their own.

Modification endpoints (update, delete, reassign, status-override) are
where the assignee/role check belongs — apply the Task Assignment
Validation Rules and the Employee restrictions above at the mutation
layer, never at the read/visibility layer.

In short: the same `GET /tasks` query is shared by every role; the
difference between roles shows up only in which buttons render and
which `PATCH`/`DELETE` requests the backend will accept — never in
which rows the `GET` returns.

SOFT DELETE POLICY
================================================================

Do not permanently delete critical records.

Use soft delete wherever possible.

Applicable entities:

- Spaces
- Lists
- Tasks
- Channels

Store:

- Deleted By
- Deleted At

Deleted records should remain available for:

- Audit Logs
- Activity Tracking
- Administrative Recovery

Every delete action must generate:

- Workspace Activity Log Entry

Activity logs must preserve historical records even when entities are
deleted.

**Implementation note for this build:** model `deletedAt` as a
nullable `DateTime?` field (not a default-empty-string field), and
ensure every read query across every service filters on
`deletedAt: null` for "active" record lookups. This is the single
highest-risk correctness rule in this entire specification — a missed
filter on any read path will resurrect soft-deleted records.

## 5C. DATA MODEL & SOCKET CONTRACT FOR SPACES AND CHAT
### (HIGH-RISK AREAS — IMPLEMENT WITH EXTRA CARE)

Spaces are the structural heart of the product (the entire task
hierarchy hangs off them) and chat is the most concurrency-sensitive
subsystem (it's the one thing that must work correctly under load with
multiple backend replicas). Both get a concrete schema and contract
here so neither is left to improvisation.

### 5C.1 Spaces — concrete Prisma shape

```prisma
model Space {
  id          String    @id @default(uuid())
  workspaceId String
  name        String
  description String    @default("")
  icon        String    @default("")
  color       String    @default("#6366F1")
  createdById String
  createdAt   DateTime  @default(now())
  updatedAt   DateTime  @updatedAt
  deletedAt   DateTime?
  deletedById String?

  workspace   Workspace @relation(fields: [workspaceId], references: [id])
  createdBy   User      @relation(fields: [createdById], references: [id])
  lists       List[]

  @@index([workspaceId])
  @@index([workspaceId, deletedAt])
}

model List {
  id          String    @id @default(uuid())
  workspaceId String
  spaceId     String
  name        String
  createdById String
  createdAt   DateTime  @default(now())
  updatedAt   DateTime  @updatedAt
  deletedAt   DateTime?
  deletedById String?

  workspace   Workspace @relation(fields: [workspaceId], references: [id])
  space       Space     @relation(fields: [spaceId], references: [id])
  createdBy   User      @relation(fields: [createdById], references: [id])
  tasks       Task[]

  @@index([workspaceId])
  @@index([spaceId, deletedAt])
}
```

**Cascade-on-soft-delete rule:** soft-deleting a Space does not
cascade-delete its Lists or Tasks in the database (no destructive
cascade). Instead, the read layer must treat a List/Task as
inaccessible if its *parent* Space (or List) has `deletedAt != null`,
even if the child row itself is still active. Implement this as a join
condition in the query layer (`WHERE space.deletedAt IS NULL`), not as
a denormalized flag copied onto every child row, to avoid drift if a
Space is ever restored.

**Space → List → Task fetch pattern:** the "Spaces & Directories"
drill-down screen (5A.5/5A.6) should be backed by three distinct,
separately-paginated endpoints — `GET /spaces`, `GET /spaces/:id/lists`,
`GET /lists/:id/tasks` — not one deeply-nested response. This keeps
each level independently cacheable and keeps payloads small as the
workspace grows.

### 5C.2 Chat — concrete Prisma shape (one model, two modes)

Per the Chat System Model's "build once, reuse" rule, Channels and
Direct Messages share one underlying `Conversation` + `Message` schema,
distinguished only by a `type` field:

```prisma
model Conversation {
  id          String    @id @default(uuid())
  workspaceId String
  type        String    // "CHANNEL" | "DIRECT"
  name        String?   // null for DIRECT conversations
  description String    @default("")
  createdById String     // the channel creator / DM initiator
  isArchived  Boolean   @default(false)
  createdAt   DateTime  @default(now())
  updatedAt   DateTime  @updatedAt
  deletedAt   DateTime?
  deletedById String?

  workspace   Workspace @relation(fields: [workspaceId], references: [id])
  members     ConversationMember[]
  messages    Message[]

  @@index([workspaceId, type])
}

model ConversationMember {
  id             String   @id @default(uuid())
  conversationId String
  userId         String
  role           String   @default("MEMBER") // "OWNER" | "ADMIN" | "MEMBER" — channel-scoped, distinct from workspace role
  joinedAt       DateTime @default(now())
  lastReadAt     DateTime?

  conversation   Conversation @relation(fields: [conversationId], references: [id])
  user           User         @relation(fields: [userId], references: [id])

  @@unique([conversationId, userId])
  @@index([userId])
}

model Message {
  id             String    @id @default(uuid())
  workspaceId    String
  conversationId String
  senderId       String
  body           String
  attachmentKeys String[]  @default([]) // MinIO object keys
  mentionedUserIds String[] @default([])
  pinnedAt       DateTime?
  createdAt      DateTime  @default(now())
  updatedAt      DateTime  @updatedAt
  deletedAt      DateTime?

  workspace      Workspace    @relation(fields: [workspaceId], references: [id])
  conversation   Conversation @relation(fields: [conversationId], references: [id])
  sender         User         @relation(fields: [senderId], references: [id])

  @@index([conversationId, createdAt])
  @@index([workspaceId])
}
```

**Why one model, not two:** a `ConversationMember.role` of `OWNER`
captures "channel creator" distinctly from the user's *workspace* role
(`SUPER_ADMIN`/`ADMIN`/`EMPLOYEE`). Channel Ownership Restrictions
(Section 5) are then enforced by checking *both*: the workspace role
(for delete/archive/manage-permissions, always gated to
`SUPER_ADMIN`/`ADMIN`) and the conversation-level `OWNER` role (for
rename/description, allowed to the creator regardless of workspace
role). For `type = "DIRECT"`, there are always exactly 2
`ConversationMember` rows and no channel-management actions apply at
all — the permission checks for delete/archive/manage-members
short-circuit to "not applicable" rather than running.

### 5C.3 Socket.io — the event contract (this must be strong, not improvised)

**Connection lifecycle:**

1. Client connects with the JWT access token (sent via the Socket.io
   `auth` payload, not a query string, to avoid leaking it into proxy
   logs).
2. Server validates the token in a Socket.io middleware
   (`io.use(...)`) **before** allowing the connection to proceed —
   an invalid/expired token must reject the connection at handshake,
   not after.
3. On successful connection, the server automatically joins the socket
   to exactly one workspace room: `workspace:{workspaceId}` (derived
   from the verified token, never from client-supplied data).
4. The server additionally joins the socket to one room per
   conversation the user is a member of:
   `conversation:{conversationId}`, computed server-side from
   `ConversationMember` rows at connect time. The client never tells
   the server which conversation rooms to join.
5. On disconnect, the presence service (5C.4) marks the user offline
   after a grace period (do not flip to "offline" instantly on a
   single dropped connection — a page refresh or brief network blip
   shouldn't flicker presence; use a 10–15 second grace window before
   marking offline).

**Event catalogue (server → client), all scoped to a room per the
naming convention in 6.3:**

| Event | Room | Payload | Trigger |
|---|---|---|---|
| `message:new` | `conversation:{id}` | full Message + sender summary | New message sent (channel or DM) |
| `message:deleted` | `conversation:{id}` | `{ messageId }` | Soft-delete of a message |
| `message:reaction` | `conversation:{id}` | `{ messageId, emoji, userId, action: 'add'\|'remove' }` | Reaction toggle |
| `conversation:typing` | `conversation:{id}` | `{ userId, isTyping }` | Typing indicator (ephemeral, not persisted) |
| `conversation:memberAdded` | `conversation:{id}` | `{ conversationId, userId }` | Added to a channel |
| `conversation:created` | `workspace:{id}` | conversation summary | New channel created (so the discoverable list updates live) |
| `task:created` / `task:updated` / `task:statusChanged` | `workspace:{id}` | full Task | Any task mutation |
| `notification:new` | `user:{userId}` *(see below)* | full Notification | Any action that generates a Unified Alert |
| `presence:update` | `workspace:{id}` | `{ userId, status: 'online'\|'offline' }` | Presence change (5C.4) |
| `member:roleChanged` | `workspace:{id}` | `{ userId, previousRole, newRole }` | Role promotion/demotion |

**Personal, per-user delivery:** notifications are the one event type
that must NOT broadcast to the whole workspace room — a Unified Alert
is private to its recipient. Maintain a per-user room
(`user:{userId}`, joined automatically at connect, same pattern as the
workspace room) and emit `notification:new` only to that room, never
to `workspace:{id}`. This is a common mistake worth calling out
explicitly: broadcasting notifications workspace-wide would leak the
existence/content of one person's alert to everyone online.

**Client → server events (the only ones the client may emit):**

| Event | Payload | Server-side validation required |
|---|---|---|
| `message:send` | `{ conversationId, body, attachmentKeys? }` | Caller must be a `ConversationMember` of `conversationId`; message is persisted via the REST-equivalent service function (not duplicated logic) before the `message:new` broadcast fires |
| `conversation:typing` | `{ conversationId, isTyping }` | Caller must be a member; never persisted, fire-and-forget |
| `presence:heartbeat` | `{}` | Refreshes the Redis presence TTL (5C.4); no broadcast on every heartbeat, only on actual state transitions |

**Critical rule:** every socket-emitted mutation (`message:send` being
the main one) must run through the *exact same* service-layer function
used by the equivalent REST endpoint — never duplicate the permission
check or the database write between the REST controller and the socket
handler. The socket handler is a thin wrapper that calls
`messageService.createMessage(...)` and then emits; it does not
reimplement validation inline. This guarantees the API Authorization
Standard (Section 5) applies identically whether a message arrives via
REST or via socket.

### 5C.4 Presence — concrete Redis shape

```
SET presence:{userId} "online" EX 20      // refreshed every ~10s by presence:heartbeat
```

- A user is "online" if `presence:{userId}` exists in Redis; "offline"
  otherwise (key expiry handles the disconnect-without-clean-close
  case automatically — no separate cleanup job needed).
- On every heartbeat, only emit `presence:update` if the state actually
  *changed* (i.e. the key didn't exist before this `SET`) — re-emitting
  on every single heartbeat to the whole workspace room at scale is
  exactly the kind of thing that makes "the socket layer weak" under
  load; emit on transitions only.
- Because this lives in Redis (not in-process memory), presence is
  correct across however many backend replicas are running — any
  replica can answer "is this user online" by checking the same key.

### 5C.5 Horizontal scaling verification for chat specifically

Beyond the general Redis-adapter requirement in Section 4.3, the build
is not done for chat until this is demonstrated, not just configured:

1. Run at least 2 backend container replicas behind Nginx.
2. Connect Client A to replica 1 and Client B to replica 2 (achievable
   in local testing by exposing each replica on a different port and
   pointing each client at a different one directly, bypassing the
   load balancer for the test).
3. Send a message from Client A in a channel both are members of.
4. Confirm Client B receives `message:new` in real time despite being
   connected to a different Node process — this is what proves the
   Redis adapter is actually wired correctly, as opposed to merely
   present in `package.json`.

### 5D.2 Controller layer — exact responsibilities per file

A controller's job, in every file below, is **only**: parse/validate
the request (via the matching Zod validator), call exactly one service
function, and shape the response into the standard envelope (6.2). A
controller must never contain a Prisma query directly — that always
belongs in the service layer.

| Controller file | Must implement |
|---|---|
| `auth.controller.js` | `signup`, `login`, `refreshToken`, `logout`. Sets/clears the httpOnly JWT cookies; never returns the raw password hash in any response. |
| `workspace.controller.js` | `createWorkspace`, `getWorkspaceSettings`, `updateWorkspaceSettings`, `deleteWorkspace` (soft delete). Every mutating action here requires `requireRole('SUPER_ADMIN')` from the middleware. |
| `invitation.controller.js` | `createInvitation`, `listPendingInvitations`, `acceptInvitation`. `createInvitation` must reject role=`SUPER_ADMIN`/`ADMIN` requests from a caller whose own role is `ADMIN` (Super-Admin-only invite targets), returning 403 before reaching the service. |
| `space.controller.js` | `createSpace`, `listSpaces`, `getSpace`, `updateSpace`, `deleteSpace` (soft). Create/update/delete gated to `SUPER_ADMIN`/`ADMIN`; `listSpaces`/`getSpace` open to all workspace members (no role gate at all — see Task/Space Visibility Implementation Requirement, same principle applies to Spaces). |
| `list.controller.js` | `createList`, `listListsForSpace`, `updateList`, `deleteList` (soft). Same read-open / write-gated pattern as spaces. |
| `task.controller.js` | `createTask`, `listTasks` (workspace-scoped, paginated, filterable per Sorting/Filtering standards), `getTask`, `updateTask`, `updateTaskStatus`, `deleteTask` (soft), `selfAssignTask`. `updateTask`/`deleteTask` must check resource ownership (caller is assignee, or caller is `ADMIN`/`SUPER_ADMIN`) inside the controller before calling the service, returning 403 on failure with the exact message pattern from 6.2's error example. |
| `comment.controller.js` | `addComment`, `listCommentsForTask`. Parses `@mentions` out of the message body server-side (do not trust a client-supplied mentioned-user list) and passes the resolved user IDs to the service so notifications fire correctly. |
| `channel.controller.js` | `createChannel`, `listChannels` (joined + discoverable, per 5A.5), `getChannel`, `updateChannelSettings`, `deleteChannel`, `archiveChannel`, `addMember`, `removeMember`. `updateChannelSettings` must branch: name/description changes allowed for the conversation's `OWNER`-role member OR workspace `ADMIN`/`SUPER_ADMIN`; `adminIds`/`memberIds` changes allowed ONLY for workspace `ADMIN`/`SUPER_ADMIN` regardless of conversation ownership (per Channel Ownership Restrictions). |
| `message.controller.js` | `sendMessage`, `listMessagesForConversation` (cursor-paginated), `deleteMessage`, `toggleReaction`, `pinMessage`. `sendMessage` must call the exact same `messageService.createMessage(...)` function the `message:send` socket handler calls (5C.3) — there is exactly one code path for message creation, entered from either REST or socket. |
| `notification.controller.js` | `listNotifications` (paginated, per-user), `markAsRead`, `markAllAsRead`. Always scoped to `recipientId = currentUser.id` — never accepts a recipient ID from the request. |
| `activityLog.controller.js` | `listActivityLogs` (paginated, filterable by Event Type / User / Date Range). Gated to `ADMIN`/`SUPER_ADMIN`; read-only, no write endpoints exposed (logs are only ever created internally by other services, never via a direct API call). |
| `member.controller.js` | `listMembers`, `changeRole`, `removeMember`, `transferOwnership`. `changeRole`/`removeMember`/`transferOwnership` must call the Last-Super-Admin-Protection check (5C/Section 5) inside the service before committing, and must reject if the target would leave the workspace with zero Super Admins. |
| `fileHub.controller.js` | `listMyFiles`, `pinFile`, `favoriteFile`, `renameFileLabel`. All scoped to `uploadedById = currentUser.id` joined against `Attachment` rows tied to tasks assigned to them — this controller never writes to the underlying `Attachment` row's `objectKey`/`taskId`, only to a personal-label/pin/favorite overlay (so the "must not duplicate the file" rule in File Ownership Model holds structurally, not just by convention). |
| `personalSpace.controller.js` | `listPersonalTasks`, `createPersonalTask`, `updatePersonalTask`, `getScratchpad`, `updateScratchpad`. Every query here filters by `userId = currentUser.id` with no workspace-role branching at all — Personal Space has exactly one permission rule ("only the owner"), so this controller has no `requireRole` middleware, only `authenticate`. |

### 5D.3 Service layer — exact responsibilities per file

Services own all Prisma access, all `$transaction` blocks, all
Activity Log writes, and all Socket.io emit calls triggered by a
mutation. Controllers call services; services never call controllers.

| Service file | Must implement |
|---|---|
| `auth.service.js` | Password hashing/verification (bcrypt), JWT signing/verification (delegates to `utils/tokens.js`), user lookup by email. |
| `workspace.service.js` | `createWorkspace` — wraps Workspace row creation + Super Admin `Membership` row + Default Workspace Bootstrap (General Space/Getting Started List/#general Conversation) + `WORKSPACE_CREATED` Activity Log entry in one `$transaction` (Database Transaction Rules). |
| `invitation.service.js` | `createInvitation` (writes the row, sends the email via `nodemailer`), `acceptInvitation` — wraps user creation (if needed) + `Membership` creation + Activity Log + notification to inviter in one `$transaction`; checks `expiresAt` and `status` before proceeding, rejecting expired/non-pending invitations. |
| `space.service.js` / `list.service.js` | Standard CRUD against `Space`/`List`, always writing `deletedAt`/`deletedById` instead of a real delete, always emitting the matching Activity Log entry on create/update/delete. |
| `task.service.js` | CRUD against `Task`. `updateTaskStatus` enforces the Task Status Validation transition table — reject any transition not in `{TODO→IN_PROGRESS, IN_PROGRESS→IN_REVIEW, IN_REVIEW→DONE}` unless the caller is `ADMIN`/`SUPER_ADMIN` (manager override). Emits `task:created`/`task:statusChanged` socket events (5C.3) and writes the matching Activity Log entry. |
| `taskAssignment.service.js` | Isolated specifically to hold the Task Assignment Validation Rules matrix (who can assign to whom) — kept separate from `task.service.js` so this one permission matrix has a single, easily-audited file rather than being buried inside general task CRUD. |
| `selfAssignment.service.js` | `selfAssignTask` — sets `assignmentType = SELF_ASSIGNED`, `createdById = assignedToId = currentUser.id`, and immediately notifies every `ADMIN`/`SUPER_ADMIN` `Membership` in the workspace (queries `Membership` for those roles, fans out a `Notification` row per recipient). |
| `comment.service.js` | Persists `Comment`, resolves `mentionedUserIds` into `Notification` rows, emits the comment-added socket event. |
| `channel.service.js` | CRUD against `Conversation`/`ConversationMember` for `type = "CHANNEL"`. Creating a channel auto-inserts the creator as a `ConversationMember` with `role = "OWNER"`. |
| `message.service.js` | `createMessage` — the single shared function called by both the REST controller and the chat socket handler (5C.3); persists `Message`, resolves mentions, and returns the full message object for the caller to broadcast. Does not itself call `io.emit` — that's the caller's job, keeping this function equally usable from a context that has no socket (e.g. a future bot/webhook). |
| `notification.service.js` | `createNotification` (used internally by every other service that needs to notify someone), `markAsRead`/`markAllAsRead`. Every `createNotification` call also triggers the `notification:new` socket emit to `user:{recipientId}` (5C.3) — this is the one centralized place that combines "write the row" and "emit the event" so no other service has to remember to do both. |
| `activityLog.service.js` | `logActivity(actorId, workspaceId, eventType, targetType, targetId, metadata)` — the single function every other service calls to write an `ActivityLog` row. No other file writes to that table directly. |
| `member.service.js` / `roleChange.service.js` | `member.service.js` handles listing/removal; `roleChange.service.js` isolates promote/demote/transfer-ownership specifically, each wrapped in a `$transaction` with the Last Super Admin Protection check run inside the same transaction (not as a separate pre-check, to avoid a race between the check and the write). |
| `fileHub.service.js` | Reads `Attachment` rows joined to the user's personal pin/favorite/label overlay table; never mutates the underlying `Attachment` row. |
| `softDelete.service.js` | One shared helper (`softDelete(model, id, deletedById)`) used by every other service's delete operations, so the soft-delete write pattern (`deletedAt`/`deletedById`) is implemented in exactly one place rather than copy-pasted across `space.service.js`, `list.service.js`, `task.service.js`, and `channel.service.js` independently. |
| `personalSpace.service.js` | CRUD against `PersonalTask`/`ScratchpadNote`, always filtered by `userId`, no workspace/role parameters accepted at all. |

### 5D.4 What "models folder" means in this stack

There is intentionally no separate `models/` folder in the backend —
Prisma's `schema.prisma` (5D.1) **is** the model layer; `prisma generate`
produces the typed client that services import directly
(`const { PrismaClient } = require('@prisma/client')`). Do not create
a parallel hand-written model layer on top of Prisma — that would be
the "second system" the Real-Time Event System rule explicitly warns
against, applied to data modeling instead of sockets. If a model needs
a computed/derived helper (e.g. "is this the workspace's last Super
Admin"), that logic lives in the relevant service file (`roleChange.service.js`
for that example), not in a models layer.

## 6. CROSS-CUTTING ENGINEERING STANDARDS

These supplement Section 5 and apply to every feature built.

### 6.1 Security
- Hash all passwords with bcrypt (cost factor 12+).
- Issue short-lived JWT access tokens (15 min) + longer-lived refresh
  tokens (7 days) stored in httpOnly, secure, sameSite cookies.
- Apply `helmet` for HTTP security headers.
- Apply CORS with an explicit allow-list (no wildcard origins in
  production).
- Sanitize all user input before storing or rendering (prevent XSS in
  chat messages, comments, task descriptions).
- Validate every request body server-side with Zod, even for fields
  the frontend already validates.
- Never trust a role or workspace ID sent from the client — always
  re-derive the caller's role and workspace from the authenticated
  session/JWT on the server.

### 6.2 API response shape
Every API response must follow one consistent envelope, matching the
Error Handling Standard in Section 5:

```json
// Success
{ "success": true, "message": "Task created", "data": { ... } }

// Failure
{ "success": false, "errorCode": "FORBIDDEN_ROLE", "errorMessage": "Only Admins and Super Admins can create spaces." }
```

### 6.3 Real-time event naming convention
Use a consistent `domain:action` naming scheme for Socket.io events,
for example: `task:created`, `task:statusChanged`, `message:sent`,
`notification:new`, `member:roleChanged`, `presence:update`. Document
every event name and payload shape in a shared constants file used by
both frontend and backend (mirror the constant values manually since
the backend is JS and frontend is TS).

### 6.4 Frontend permission helpers
Build a single `usePermissions()` hook on the frontend that exposes
boolean checks (e.g. `canManageSpaces`, `canAssignTasks`,
`canAccessWorkspaceSettings`) derived from the current user's role.
Every UI gate in the app should call this hook — never inline a raw
`role === 'ADMIN'` check scattered across components.

### 6.5 Backend permission helpers
Mirror this on the backend with a single `requireRole(...)` middleware
factory and a `permissions.js` utility module containing the same
logical checks, so the business rules in Section 5 are implemented in
exactly one place and reused by every controller.

## 8. DEFINITION OF DONE

A feature or the application as a whole is not complete until:

- [ ] Every permission rule in Section 5 is enforced server-side, not
      just hidden in the UI.
- [ ] Every entity listed in the Workspace Data Isolation Rules and
      Entity Ownership Model carries a `workspaceId` and the correct
      ownership fields, and every query filters by them.
- [ ] Soft delete is implemented with nullable `deletedAt`/`deletedById`
      and every "active records" read path filters it out.
- [ ] The Last Super Admin Protection is implemented and tested (attempt
      to demote/remove/transfer away the sole Super Admin and confirm
      it's blocked).
- [ ] Every mutating action that should produce an Activity Log entry
      does so, with actor, workspace, and event type recorded.
- [ ] All real-time events are scoped to `workspace:{workspaceId}`
      Socket.io rooms — verified by confirming a user in one workspace
      never receives an event from another.
- [ ] File uploads go through MinIO, never local disk, and task/chat
      attachments correctly populate the recipient's File Hub without
      duplication.
- [ ] API responses follow the consistent success/failure envelope
      everywhere.
- [ ] The app runs fully via `docker-compose up --build`, bringing up
      frontend, backend, Postgres, Redis, and MinIO together.
- [ ] Basic load testing confirms multiple concurrent Socket.io
      connections and REST requests are served correctly across at
      least 2 backend replicas behind the Nginx reverse proxy.
- [ ] Every sidebar item for every role passes all four checks in the
      "No Patches" Completeness Rule (Section 5A.9): correct
      visibility, real data-backed screen, working backend-wired
      actions, working pagination/sorting/filtering.
- [ ] Every row in the ClickUp Feature-Parity Checklist (Section 5B)
      is implemented and demonstrable end-to-end — from Super Admin
      signup and workspace creation, through inviting an Admin and an
      Employee, to that Employee using Tasks, Chat, and their Personal
      Space.
- [ ] No screen in the application is a placeholder, "Coming Soon"
      notice, or otherwise non-functional stub.
- [ ] The Space → List → Task cascade-on-soft-delete rule (5C.1) is
      verified: soft-deleting a Space hides its Lists and Tasks from
      every read path without destructively deleting the underlying
      rows.
- [ ] Channels and Direct Messages are confirmed to share one
      `Conversation`/`Message` schema (5C.2), not two parallel
      implementations.
- [ ] The full Socket.io event catalogue in 5C.3 is implemented, with
      every socket-emitted mutation routed through the same service
      function as its REST equivalent — never a duplicated/divergent
      permission check.
- [ ] Notifications are confirmed to broadcast only to the recipient's
      personal `user:{userId}` room, never to the workspace-wide room.
- [ ] The multi-replica chat verification in 5C.5 has been performed
      and a message sent by a client on one backend replica is
      confirmed received in real time by a client on a different
      replica.
- [ ] Every Prisma model in 5D.1 exists in `schema.prisma` with the
      listed fields and indexes; no controller file contains a direct
      Prisma query (all database access lives in the service layer,
      per 5D.2/5D.3); no hand-written `models/` layer was added beside
      Prisma (5D.4).
